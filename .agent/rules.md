# 작업 원칙 및 배포 가이드라인

본 저장소의 모든 작업 시 다음의 원칙을 반드시 준수한다.

## 1. 일일 첫 작업 전 동기화 (Daily First-work Sync)
- **하루 기준 처음** 공유 받은 작업이나 프롬프트 입력을 수행할 때, 원격 저장소와의 동기화를 확인한다. (매 프롬프트마다 수행할 필요는 없음)
- `git pull`을 통해 원격 저장소의 최신 변경 사항을 로컬로 가져온다.
- `/시작` 워크플로우를 활용하여 일자별 작업 시작 시 동기화를 처리한다.

## 2. 배포 시 버전 업데이트 (Deployment Versioning)
- 코드를 배포(Push)할 때마다 사이트의 버전 번호를 업데이트한다.
- **대상 파일**: `src/components/SiteFooter.vue` 내의 `Version` 정보.
- **업데이트 규칙**: 
  - **버전 형식**: 반드시 **소수점 한 자리**까지만 사용한다 (예: 21.4 -> 21.5). 소수점 두 자리 이하(21.4.1 등)는 절대 사용하지 않는다.
  - 단순 버그, 스타일 수정, 기능 추가 등 모든 일반 작업: `Minor` 버전 업데이트 (예: 21.4 -> 21.5)
  - **대규모 구조적 변경 및 메이저 업데이트**: (예: 21.x -> 22.0) **(반드시 사용자에게 확인 후 업데이트)**
- **절차**: 코드 수정 완료 후 `SiteFooter.vue`의 버전을 규칙에 따라 조정한 뒤 한꺼번에 커밋하여 푸시한다.

## 3. UI 및 사용자 경험 원칙
- **스켈레톤 유지**: 데이터 로딩 중이거나 결과가 없을 때, 임의로 "소식이 없습니다"와 같은 메시지를 추가하지 않는다. 사용자의 요청이 없는 한 기존의 **스켈레톤(Skeleton) 로딩 상태를 유지**한다.
- **임의 변경 금지**: 기능 개선이라는 명목으로 기획에 없는 UI 요소나 안내 문구를 마음대로 추가하지 않는다.

## 4. 커밋 메시지 규칙
- 작업 내용에 따라 적절한 접두어를 사용한다 (예: `feat:`, `fix:`, `style:`, `chore:`).
- 버전 업데이트가 포함된 경우 커밋 메시지에 명시한다 (예: `fix: blog loading and bump version to 21.5`).

## 5. 작업 시작 전 필수 사항
- **매 작업 시작 전** 반드시 `.agent/rules.md` 파일을 로드하여 모든 원칙을 숙지한다.
- 사용자의 이전 요청 사항(예: 스켈레톤 유지 등)을 최우선으로 준수한다.
